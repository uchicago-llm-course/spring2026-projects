# Week Six Progress Update

## Preprocessing
We have built a preprocessing pipeline that turns the generated chess and variant game archives into model-ready token chunks, while preserving enough structure to support the training curriculum. The ordinary path reads the raw generated shards, validates them, tokenizes games into fixed-length training sequences, writes chunked .npz outputs, and records manifests/reports so we can audit what was consumed. It supports standard games, local NAS variant games, and downloaded Modal variant games, with the known bad shard explicitly excluded. The current ordinary processed corpus is roughly 10B tokens before composed-game augmentation.
The preprocessing also builds position indexes for composition. For each game, it hashes intermediate board positions and stores compact references back to the source game and ply. That lets us find games that pass through the same position and splice them together into “composed games.” This is important for the project because the model is not told the active ruleset directly; it only has the loaded module/core context. Composed games create pressure for the model to infer and preserve piece-specific behavior across shared positions, including transitions from standard to variant play, variant to standard play, variant to variant play, and no-rule-change compositions. We also added progress logging, bounded memory buffering, shard exclusion, sampled standard indexing, and parallelized indexing/composition so the full run can execute detached under tmux.

## Model
On the model side, we built a modular chess transformer scaffold around the idea that different piece/rule modules can be loaded without simply giving the model an explicit ruleset ID. The architecture is meant to separate shared chess reasoning from piece-specific behavior: shared layers handle general sequence modeling and board/game structure, while modular cores specialize around rule or piece behavior. That matches the research goal: variants are not primarily there because we care about variant playing strength, but because they create training pressure that should encourage the model to localize “how pieces work” into the relevant modules instead of smearing that knowledge through the shared network.

## Training Scaffold
The training scaffold ties this together with a curriculum rather than just shuffling all data uniformly. The proposed schedule keeps standard chess present throughout, with a floor around 15-20% so the final model remains competent at normal chess. Early training emphasizes standard and simple single-piece variants; later phases add same-piece families, then two- and three-piece variants, then a full balanced mix including composed games. We also discussed a later training strategy where some backward passes freeze shared weights and only update relevant cores, to further discourage the model from encoding piece-specific rules in the shared layers.
The current remaining bottleneck is not conceptual coverage but preprocessing throughput, especially composition. Ordinary tokenization and position indexing are working; composition is running but still too slow and underutilizing the machine. The pipeline now has the right pieces: validated sources, token chunks, position hashes, composed-game generation, curriculum-aware sampling hooks, manifests, reports, and resumable detached execution. The next engineering focus is making composition fast enough to produce the composed portion of the 35B-token target on the local NAS without wasting CPU or memory.
